# 仕組み (やさしい解説)

専門用語をできるだけ避けて、Smart Shield の中で起きていることを順を追って説明します。

## 全体像

```
                ┌───────────────────────────────────────────────────┐
                │                  サーバ (Linux/Windows)             │
                │                                                   │
   攻撃者 ──→ │  ① SSH / ウェブサーバが「失敗した」とログに書く      │
                │                       │                           │
                │                       ▼                           │
                │  ② Smart Shield がログを 1 行ずつ読む               │
                │     (ファイルを追いかける、いわゆる tail)            │
                │                       │                           │
                │            ┌──────────┴──────────┐                │
                │            ▼                     ▼                │
                │  ③a ルールエンジン          ③b AI エンジン           │
                │  (パターン一致で即時)     (数分ごとに Claude へ)      │
                │            │                     │                │
                │            └──────────┬──────────┘                │
                │                       ▼                           │
                │  ④ 怪しい IP を判定 → ⑤ OS のファイアウォールに      │
                │                          ブロック登録               │
                │                                                   │
                │  ⑥ 一定時間経ったら ⑦ 自動でブロック解除            │
                └───────────────────────────────────────────────────┘
```

## ステップごとの説明

### ① サーバが「失敗」をログに書く

普段使っているサーバソフト (OpenSSH, Nginx, Apache など) は、
**誰がいつ何にアクセスし、成功・失敗したか** を勝手に記録しています。例:

```
May 28 10:16:11 host sshd[1011]: Failed password for invalid user admin from 203.0.113.7 port 51001 ssh2
203.0.113.99 - - [28/May/2026:11:00:01 +0000] "GET /.env HTTP/1.1" 404 162 "-" "curl/7.83"
```

1 行目は「SSH に admin というユーザでログインしようとしてパスワードが間違っていた」、
2 行目は「`.env` という秘密の設定ファイルを盗もうとしてアクセスした (見つからず 404)」という意味です。

### ② Smart Shield がログを追いかける

Smart Shield は **常にファイルの末尾を見続けている** 状態を保ちます (`tail -f` という Unix の昔からの仕組みと同じ)。新しい行が書かれた瞬間に拾い上げます。

ファイルが入れ替わったり (ログローテーション) サーバが再起動しても、
どこまで読んだかを SQLite というデータベースに覚えているので、二重に処理したり読み飛ばしたりしません。

### ③a ルールエンジン: パターンで即時判定

代表的な攻撃の文章は **形が決まっています**。たとえば SSH の失敗は必ず

```
Failed password for ... from <IPアドレス> port ...
```

Apache の認証失敗なら

```
[client 1.2.3.4:5678] AH01617: user admin: authentication failure for ...
```

という形になります。Smart Shield は「この形に当てはまる行が、同じ IP から **10 分以内に N 回出たらブロック**」というルールを持っています。
このしきい値は `config/shield.yaml` で自由に調整できます。

同梱フィルタ: **sshd / nginx-auth / apache-auth** (Apache は access_log と error_log の両方をカバー)。

| 設定項目 | 意味 |
|---|---|
| `max_retries` | 何回失敗したらブロックするか |
| `findtime_seconds` | 何秒以内の失敗を数えるか (時間窓) |
| `ban_seconds` | 何秒間ブロックするか |

ルールエンジンの判断は **一瞬で終わる** ので、攻撃の最中にリアルタイムで止められます。

### ③b AI エンジン: 数分ごとにまとめて判定

ルールエンジンには限界があります:

- 1 回ずつしか試さない攻撃 (例: 5 分おきに 1 回ずつパスワードを試す)
- 新種の攻撃ツールが使う、まだ知られていないパターン
- 複数の動作が組み合わさってはじめて怪しいケース

こうしたケースは「行単位のパターン一致」では捉えにくいので、Smart Shield は
**数分に 1 回、直近のログをまとめて Claude (AI) に読ませて** 判定を頼みます。

Claude には「次の方針で読んでください」というルール (SOC アナリストのつもりで読む、誤検知は重大なコストなので確証なきものは出さない、私的 IP は除外、など) を渡してあります。

返ってくる答えは厳密に決まった形式 (JSON) で、

```json
{
  "blocks": [
    { "ip": "203.0.113.7", "reason": "...", "confidence": 0.92 }
  ]
}
```

Smart Shield は `confidence` (確信度) が一定値以上のものだけ採用します。
詳しくは [ai-analysis.md](ai-analysis.md) を参照。

### ④ 怪しい IP を判定

ルール / AI どちらの経路でも、最終的に出てくるのは「ブロックすべき IP のリスト」です。
このとき次のチェックが入ります:

- **ホワイトリストに該当しないか** (自分の事務所 IP / 社内ネットなど)
- **すでにブロック中ではないか** (重複登録を避ける)

これを通ったものだけがブロック対象になります。

### ⑤ OS のファイアウォールに登録

Smart Shield 自身は通信を遮るソフトではなく、OS が標準で持っているファイアウォールに
「この IP からの通信は捨ててください」と頼みます。

| OS | 使うコマンド |
|---|---|
| Linux | `iptables -I INPUT -s <ip> -j DROP` |
| Windows | `netsh advfirewall firewall add rule ...` |

`dry_run: true` (お試しモード) の間は **このコマンドは打たれません**。SQLite に「もし本番だったらこのタイミングでブロックしていた」という記録だけ残ります。

### ⑥ ⑦ 一定時間で自動解除 (永久 ban は例外)

通常のブロックには有効期限がついています。Smart Shield は 30 秒ごとに「有効期限切れの ban」を探し、
見つかったら自動でファイアウォールのルールを外します。

これは大事な仕組みで、

- 誤検知 (うっかりブロックしてしまった) があっても永久ロックされない
- 動的 IP を使う一般ユーザが、同じ IP を別の人が使い始めても巻き添えにならない

を意味します。

**ただし永久 ban (`expires_at = 0`) は自動解除の対象外** です。手動で `unban` するまでは外れません。
永久 ban になる経路は次の 3 つ:

1. CLI で `--permanent` を付けて手動 ban
2. 設定ファイルで `ban_seconds: 0` を指定
3. 再犯エスカレーション (recidive): 過去 N 日以内に M 回 ban された相手を自動的に永久 ban に昇格

### ⑧ サーバ再起動への対応

`iptables` などのファイアウォールルールは OS 再起動で消えてしまいます。
Smart Shield は **起動時に SQLite から有効な ban をすべて読み出し、OS に再登録** します。
これにより一時 ban も永久 ban もサーバ再起動を跨いで効き続けます。

## どこに何が保存されているか

| 場所 | 何が入っているか |
|---|---|
| `var/shield.sqlite3` | 失敗の記録、ブロック中の IP、過去のイベント、tail のオフセット |
| `var/shield.pid` | 起動中のプロセス番号 (二重起動の検知用) |
| systemd journal / NSSM ログ | 標準出力 (どの IP をどの理由でブロックしたか) |

SQLite はテキストでは無いですが、`sqlite3 var/shield.sqlite3` で開けば普通に SELECT で中身を見られます。
監査資料を作るときに重宝します。

## なぜこの設計なのか

- **ルール + AI の二段構え**: 高速で枯れたルール判定だけだと取りこぼしが出る、AI だけだと反応速度とコストが問題。両方が補完しあう。
- **ブロックは OS のファイアウォールに任せる**: Smart Shield 自身がパケット処理をしないので、軽量で、Smart Shield が落ちてもブロックは効いたまま。
- **判定経路は別だが ban の記録は 1 つ**: 後から「これは AI 由来? ルール由来?」を SQL で集計できる (`bans.source` カラム)。
- **dry_run モード**: 初日に本番でいきなりブロックしないで済むのが現場で一番効く安心装置。
